Introduction

ErlMUD is a game system written in Erlang. My purpose in writing it and this commentary is to demonstrate how a simple Erlang project evolves to become a production-style OTP system. (Playing games on it is fun, too!)

I don't cover any basics of the Erlang language here, Learn You Some Erlang and Programming Erlang already do a better job of that than I ever could. If you are new to Erlang it is probably a good idea to work through the parts of those books that cover the language before tackling anything beyond the first or second parts of this commentary. If you are familiar with Erlang but don't really understand what OTP is about, or understand how to write a simple OTP application but wonder what to look out for when deploying production systems then ErlMUD is for you.

Erlang the language is fairly simple (yay!), but Erlang the system is not. Understanding the fundamentals of concurrency and message protocols, learning large new libraries, working around the details of a new runtime system, using new profiling tools, remembering the quirks of release building, and utilizing much-talked-about-but-little-used hot code loading features is anything but easy.

Language tutorials and elementary resources that walk the reader through toy systems are sufficient to demonstrate the language and some even show how to build small releases. They do not give much guidance when it comes to building a production system that will require ongoing maintenance and feature upgrades, though. This is partly because the subject of production Erlang is broad and the "right" answer to any given question is usually "it depends", and partly because there is little agreement about the difference between "toy" and "production" Erlang system structure. The fastest way to grok the difference between a relatively simple system written in raw Erlang and a production OTP system is to follow a single project from infancy to adulthood.

No claim is made here that ErlMUD embodies the one true way to design or structure a production system. I'm not even claiming that ErlMUD reflects a good way to do this. What is presented is merely a way and hopefully the circuitous path we take to our destination will place enough facets of the production design problem in view of readers that they can come up with their own solutions when they encounter similar situations.

Reading code can be hard. Understanding interactions within a software system can be hard, especially a concurrent one. I think the best way to approach learning a new environment is to stick with the same project concept the whole way through so the reader isn't spending more time understanding how the project works than the environment. To relieve the reader of the burden of understanding new project mechanics every chapter I've instead decided to make the entire demonstration center around just ErlMUD so we can describe the way the game system works just once, and then focus on how the imlpementation of it evolves over time.
